# Contributing to Openflow

Thanks for contributing. Openflow is founder-led and vision-driven. Contributions are welcome, and maintainers are responsible for long-term direction.

## Before Opening a PR

1. Search existing issues first.
2. Discuss major features before building them.
3. Keep PRs focused (one feature/fix per PR).
4. Add tests when applicable.
5. Follow existing coding style.
6. Update documentation if behavior changes.

## Pull Request Rules

Every PR should include:

- What changed
- Why it changed
- Screenshots (if UI changes)
- Related issue number

Please keep PRs under about 500 lines when practical.

## Issue Rules

Use the issue templates.

Bug reports should include:

- Expected behavior
- Actual behavior
- Steps to reproduce
- Environment
- Screenshots/logs

Feature requests should include:

- Problem
- Proposed solution
- Alternatives considered

## AI-Assisted Contributions

AI-assisted contributions are welcome, but contributors are responsible for understanding, testing, and maintaining the code they submit.

## Coding Standards

- TypeScript only
- ESLint must pass
- Prettier formatting
- No commented-out code
- No unused dependencies
- Meaningful commit messages

## Project Philosophy

- Performance over abstraction
- Simplicity over cleverness
- Readability over brevity
- Backwards compatibility when reasonable
- Ship fast, iterate faster

## Maintainer Rights

Maintainers reserve the right to reject any contribution that does not align with the project vision, roadmap, or quality standards.

## Community Expectations

Contributors are encouraged to:

- Review PRs
- Improve documentation
- Answer discussions
- Report bugs
- Respect maintainers' time
